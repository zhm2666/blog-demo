# Gin-Vue-Blog 权限隔离说明文档

## 一、项目权限模型概述

Gin-Vue-Blog 项目采用 **RBAC (Role-Based Access Control) 基于角色的访问控制** 模型实现权限隔离。这是一种广泛应用于企业级系统的权限管理方案。

### 1.1 什么是 RBAC？

RBAC 的核心思想是：**将权限与角色关联，再将角色分配给用户**

```
用户（谁）    角色（职位）    权限（能做什么）
  │              │              │
  ▼              ▼              ▼
 张三         管理员         查看文章
 李四    →    编辑    →     编辑文章
 王五         访客          评论
```

## 三、后端权限实现

### 3.1 中间件架构

后端使用 **两层中间件** 实现权限控制：

```
请求 ──▶ JWTAuth() ──▶ PermissionCheck() ──▶ Handler
         (认证)        (权限验证)
```

#### 第一层：JWT 认证中间件

文件位置：`gin-blog-server/internal/middleware/auth.go`

```go
// 基于 JWT 的授权中间件
func JWTAuth() gin.HandlerFunc {
    return func(c *gin.Context) {
        db := c.MustGet(g.CTX_DB).(*gorm.DB)
        
        // 1. 获取请求的 URL 和方法
        url, method := c.FullPath()[4:], c.Request.Method
        
        // 2. 检查是否是匿名资源（无需登录即可访问）
        resource, err := model.GetResource(db, url, method)
        if err != nil {
            // 资源不存在，跳过验证
            if errors.Is(err, gorm.ErrRecordNotFound) {
                c.Set("skip_check", true)
                c.Next()
                return
            }
            handle.ReturnError(c, g.ErrDbOp, err)
            return
        }

        // 3. 匿名资源直接放行
        if resource.Anonymous {
            c.Set("skip_check", true)
            c.Next()
            return
        }

        // 4. 验证 JWT Token
        authorization := c.Request.Header.Get("Authorization")
        if authorization == "" {
            handle.ReturnError(c, g.ErrTokenNotExist, nil)
            return
        }

        // 解析 Token
        claims, err := jwt.ParseToken(g.Conf.JWT.Secret, parts[1])
        if err != nil {
            handle.ReturnError(c, g.ErrTokenWrong, err)
            return
        }

        // 5. 检查 Token 是否过期
        if time.Now().Unix() > claims.ExpiresAt.Unix() {
            handle.ReturnError(c, g.ErrTokenRuntime, nil)
            return
        }

        // 6. 获取用户信息
        user, err := model.GetUserAuthInfoById(db, claims.UserId)
        if err != nil {
            handle.ReturnError(c, g.ErrUserNotExist, err)
            return
        }

        // 7. 设置到 Context
        session := sessions.Default(c)
        session.Set(g.CTX_USER_AUTH, claims.UserId)
        session.Save()
        c.Set(g.CTX_USER_AUTH, user)
    }
}
```

#### 第二层：权限验证中间件

```go
// 资源访问权限验证
func PermissionCheck() gin.HandlerFunc {
    return func(c *gin.Context) {
        // 1. 如果跳过检查，直接放行
        if c.GetBool("skip_check") {
            c.Next()
            return
        }

        db := c.MustGet(g.CTX_DB).(*gorm.DB)
        auth, err := handle.CurrentUserAuth(c)
        if err != nil {
            handle.ReturnError(c, g.ErrUserNotExist, err)
            return
        }

        // 2. 超级管理员直接放行（无需权限检查）
        if auth.IsSuper {
            slog.Debug("[middleware-PermissionCheck]: super admin no need to check, pass!")
            c.Next()
            return
        }

        // 3. 获取请求的 URL 和方法
        url := c.FullPath()[4:]
        method := c.Request.Method

        // 4. 遍历用户的每个角色，检查是否有权限
        for _, role := range auth.Roles {
            pass, err := model.CheckRoleAuth(db, role.ID, url, method)
            if err != nil {
                handle.ReturnError(c, g.ErrDbOp, err)
                return
            }
            if !pass {
                // 没有权限，返回错误
                handle.ReturnError(c, g.ErrPermission, nil)
                return
            }
        }

        c.Next()
    }
}
```

## 四、前端权限实现

### 4.1 路由守卫

文件位置：`gin-blog-admin/src/router/guard.js`

```javascript
// 路由前置守卫
router.beforeEach(async (to) => {
    const { token } = useAuthStore()

    // 1. 没有 Token
    if (!token) {
        // 允许访问登录页和404页
        if (['/login', '/404'].includes(to.path)) {
            return true
        }
        
        // 提示并重定向到登录页
        window.$message.error('没有 Token，请先登录！')
        return { name: 'Login', query: { ...to.query, redirect: to.path } }
    }

    // 2. 有 Token 时无需访问登录页
    if (to.name === 'Login') {
        window.$message.success('已登录，无需重复登录！')
        return { path: '/' }
    }

    // 3. 路由存在则正常访问
    if (router.getRoutes().find(e => e.name === to.name)) {
        return true
    }

    // 4. 路由不存在，返回404
    return { name: '404', query: { path: to.fullPath } }
})
```

### 4.2 动态路由生成

文件位置：`gin-blog-admin/src/store/modules/permission.js`

```javascript
// 后端控制路由模式
async generateRoutesBack() {
    // 1. 调用后端接口获取当前用户的菜单
    const resp = await api.getUserMenus()
    
    // 2. 将后端菜单数据转换为前端路由格式
    this.accessRoutes = buildRoutes(resp.data)
    
    return this.accessRoutes
}

// 构建路由函数
function buildRoutes(routes = []) {
    const result = []
    
    for (const e of routes) {
        if (e.is_catalogue) {
            // 目录类型菜单
            result.push({
                name: `${e.name}Parent`,
                path: '/',
                component: shallowRef(Layout),
                isHidden: e.is_hidden,
                isCatalogue: true,
                redirect: e.redirect,
                children: [{
                    name: e.name,
                    path: e.path,
                    component: vueModules[`/src/views${e.component}/index.vue`],
                    meta: {
                        title: e.name,
                        icon: e.icon,
                        keepAlive: e.keep_alive,
                    },
                }],
            })
        } else {
            // 非目录类型菜单
            result.push({
                name: `${e.name}Parent`,
                path: e.path,
                component: shallowRef(Layout),
                children: e.children?.map(ee => ({
                    name: ee.name,
                    path: ee.path,
                    component: vueModules[`/src/views${ee.component}/index.vue`],
                })),
            })
        }
    }
    
    return result
}
```

## 五、权限控制层级

### 5.1 三层权限控制

| 层级 | 控制位置 | 控制粒度 | 实现方式 |
|------|----------|----------|----------|
| **第一层** | 后端中间件 | 接口权限 | 精确到每个 API 的 GET/POST/PUT/DELETE |
| **第二层** | 后端中间件 | 菜单权限 | 根据用户角色返回对应菜单 |
| **第三层** | 前端路由守卫 | 页面权限 | Token 验证 + 路由存在性检查 |

### 5.2 匿名资源

在 `resource` 表中，`anonymous = 1` 表示该接口可以匿名访问，无需登录：

```sql
-- 登录接口可以匿名访问
INSERT INTO resource VALUES (NULL, '/login', 'POST', '用户登录', 1);

-- 文章列表需要登录后才能访问
INSERT INTO resource VALUES (NULL, '/article/list', 'GET', '文章列表', 0);
```

## 六、超级管理员

### 6.1 超级管理员特权

数据库 `user_auth` 表中的 `is_super` 字段：
- `is_super = 1`：超级管理员
- `is_super = 0`：普通用户

超级管理员的特权：
1. **绕过所有权限检查**：不检查角色-资源关联
2. **拥有所有菜单权限**：可访问所有菜单
3. **绕过所有接口限制**

### 6.2 设置超级管理员

```sql
-- 只能在数据库中手动设置，不能通过前端修改
UPDATE user_auth SET is_super = 1 WHERE id = 1;
```

## 七、完整权限流程示例

### 场景：普通用户访问删除文章接口

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           请求流程                                          │
└─────────────────────────────────────────────────────────────────────────────┘

1. 用户发送请求
   DELETE /api/admin/article/123
   Header: Authorization: Bearer xxx.xxx.xxx

2. 【第一层】JWT 认证中间件 (JWTAuth)
   ├── 检查资源是否需要验证
   │   └── 查询 resource 表，找到 /article/:id DELETE
   │       └── anonymous = 0，需要验证
   ├── 验证 Token 是否有效
   │   ├── 解析 JWT Token
   │   ├── 检查是否过期
   │   └── 获取用户 ID = 2
   └── 获取用户信息，存入 Context

3. 【第二层】权限验证中间件 (PermissionCheck)
   ├── 从 Context 获取用户信息
   │   └── user_auth_id = 2, roles = ['管理员']
   ├── 检查是否是超级管理员
   │   └── is_super = 0，不是
   ├── 遍历用户的角色
   │   └── 角色1: 管理员
   │       └── 查询 role_resource 表
   │           └── 查找 role_id=1, resource_id 对应的 url=/article/:id, method=DELETE
   │               └── 找到，有权限
   └── 验证通过，放行

4. 执行 Handler 处理业务逻辑
   └── handle_article.go 的 Delete() 方法

5. 返回响应
   └── { code: 0, msg: "success", data: null }
```

## 八、数据库权限表详细说明

### 8.1 resource 表（接口资源）

```sql
CREATE TABLE `resource` (
  `id` bigint NOT NULL AUTO_INCREMENT,
  `parent_id` bigint NULL DEFAULT NULL,     -- 父资源ID，用于树形结构
  `url` varchar(255) NULL DEFAULT NULL,     -- 接口路径，如: /article/list
  `method` varchar(10) NULL DEFAULT NULL,   -- 请求方法: GET/POST/PUT/DELETE
  `name` varchar(50) NULL DEFAULT NULL,     -- 资源名称，如: 文章列表
  `anonymous` tinyint(1) NULL DEFAULT NULL, -- 是否匿名访问: 0=需要验证, 1=不需要
  PRIMARY KEY (`id`)
);
```

示例数据：

## 九、安全机制

### 9.1 Token 安全

| 安全措施 | 说明 |
|----------|------|
| JWT 加密 | 使用 HS256 算法 |
| 密钥存储 | 保存在配置文件中 |
| 过期时间 | 可配置（默认 24 小时） |
| 双重验证 | JWT + Session |

### 9.2 密码安全

- **加密方式**：Bcrypt Hash
- **不可逆加密**：无法从密文还原明文
- **盐值随机**：每次加密使用不同的盐
